# PRD：本地桌面端儿童字母打字小游戏 V1

- 文档状态：Draft
- 版本：V1
- 日期：2026-03-16
- 目标产品：本机运行的桌面端网页小游戏

## Background

本需求面向 5 岁、刚开始学习键盘输入的小朋友，目标是在家里的桌面电脑上提供一个足够简单、可立即开始使用的字母打字练习工具。首版只聚焦一个最小可用玩法：屏幕一次显示一个小写字母，孩子按实体键盘上的对应按键，系统即时反馈对错，并在一轮练习中展示分数。

现有已确认范围强调“低复杂度、单机、本地、无账号、无历史记录”。因此首版不追求学习系统完整性，而优先保证核心练习链路清晰、稳定、适合儿童理解和操作。

## Goals

- 提供一个可在本地桌面浏览器中运行的单机网页小游戏。
- 通过“一次一个小写字母”的方式完成 26 个英文字母基础键盘练习。
- 对每次输入提供即时、清晰、儿童可理解的对错反馈。
- 在单轮练习中实时显示分数，并在结束时展示结果。
- 将交互复杂度控制在 5 岁儿童可独立使用或在家长陪同下轻松使用的水平。

## Scope

- 本地运行，不依赖后端服务、账号系统或网络连接。
- 仅支持桌面端电脑和实体键盘输入。
- 练习内容仅覆盖 26 个小写英文字母 `a-z`。
- 首版仅提供一种核心玩法：单个字母逐个出现，用户按键匹配。
- 练习过程中提供即时视觉反馈。
- 单轮练习中展示当前分数，并在结束后展示结果页。
- 提供重新开始当前玩法的能力。

## Non-Goals

- 不支持大写字母、数字、符号、单词或句子练习。
- 不支持掉落字母、连续跟打、多模式切换或关卡系统。
- 不做账号、登录、云同步、联网对战或任何在线功能。
- 不做历史记录、学习报告、成长曲线、成就系统或长期进度追踪。
- 不优先支持手机、平板、触屏或手柄输入。
- 不做多用户切换、家长后台或复杂配置系统。

## Assumptions / Scope Notes

- 轮次结束规则目前未锁定。实现上应采用“本地可配置”的结束机制，但本 PRD 不预设固定题量或固定时长。
- 声音反馈是否加入尚未确认。当前将“视觉反馈”定义为 P0 必需项，声音反馈不纳入当前上线验收基线。
- 界面视觉风格尚未确认。当前仅约束为“高可读、大字号、高对比、低干扰”，不锁定“卡通”或“极简”的具体设计方向。
- 为降低儿童误操作成本，按键匹配建议按字母本身比较，忽略 `Shift` 或 `Caps Lock` 带来的大小写差异。
- 在计分规则未单独确认前，首版默认分数定义为“本轮正确完成的字母数”。
- 在错误处理规则未单独确认前，首版默认错误后保留当前字母，直到输入正确或轮次结束条件触发。

## User Scenarios

1. 小朋友在家长帮助下打开本地网页，点击“开始练习”，立即进入游戏，无需登录或设置。
2. 游戏界面中央显示一个大号小写字母，例如 `m`。小朋友按下键盘上的 `m`，系统立即显示正确反馈，分数增加，并切换到下一个字母。
3. 小朋友按错按键时，系统立即显示错误反馈，当前字母保持不变，等待再次输入。
4. 小朋友完成一轮练习后，看到本轮分数和明显的“再来一轮”入口，可以重新开始。
5. 家长陪同观察时，可以通过当前分数和孩子的即时反馈状态快速理解孩子这一轮的表现，而无需查看历史记录。

## Functional Requirements

### FR-01 启动与进入

- 产品必须可在本地桌面浏览器中运行。
- 产品不得要求登录、注册、联网或后端初始化。
- 首屏必须提供一个清晰、单一的主操作入口，例如“开始练习”。
- 启动后应直接进入当前唯一玩法，不出现模式选择页面。

### FR-02 轮次状态管理

- 系统至少支持三个状态：开始页、练习中、结果页。
- 练习开始时，系统应初始化当前分数、当前题目和反馈状态。
- 练习结束条件必须由本地配置控制，以便后续在不改动主流程的前提下调整规则。
- 结果页必须支持重新开始一轮，重新开始后所有本轮状态重置。

### FR-03 出题与展示

- 练习中界面任一时刻只能展示一个目标字母。
- 目标字母范围必须严格限制为 26 个小写英文字母 `a-z`。
- 不得出现大写字母、数字、符号或组合字符作为目标内容。
- 目标字母必须位于视觉焦点区域，字号和对比度应明显高于其他界面元素。

### FR-04 输入处理

- 系统必须监听实体键盘输入，并在练习状态下处理 `a-z` 对应按键。
- 字母比较应采用大小写归一化处理，例如等价于 `event.key.toLowerCase()` 的比较逻辑。
- 非字母按键输入不得导致报错、卡死或异常跳题。
- 当页面失去焦点导致无法接收输入时，系统应给出可见提示，而不是静默失效。

### FR-05 对错反馈

- 每次有效输入后，系统必须立即给出反馈。
- 正确输入时，系统必须展示明确的正确反馈，并更新分数。
- 正确输入后，系统应在短暂反馈后自动切换到下一个字母。
- 错误输入时，系统必须展示明确的错误反馈。
- 在当前默认规则下，错误输入后当前字母保持不变，等待下一次输入。
- 反馈方式以视觉为必需项，且不应依赖儿童阅读长文本才能理解。

### FR-06 分数与结果

- 练习过程中必须持续显示当前分数。
- 在当前默认规则下，分数应等于本轮已正确完成的字母数量。
- 结果页必须展示本轮最终分数。
- 结果页必须提供重新开始入口。
- 系统不得保存跨轮次历史分数或长期学习记录。

### FR-07 交互与界面简化

- 主界面必须保持单任务焦点，不展示与当前练习无关的复杂信息。
- 可点击元素必须数量少、层级浅，避免儿童误触后进入复杂流程。
- 页面说明文案应尽量短，必要时可通过颜色、图形或动效辅助理解。
- 首版不得加入会干扰按键练习注意力的复杂动画或装饰性流程。

### FR-08 本地与数据约束

- 系统不得发起任何业务必需的网络请求。
- 系统不得要求用户创建身份信息。
- 系统不得记录个人资料、设备标识、历史行为或学习轨迹。
- 关闭页面或刷新页面后，不要求保留上一次练习状态。

## Non-Functional Requirements

- 必须支持离线使用，断网情况下仍可完成完整练习流程。
- 最低验收运行环境为主流桌面浏览器，基线建议为最新稳定版 Chrome。
- 页面首次可交互时间应控制在家庭常见电脑可接受范围内，目标不超过 3 秒。
- 从有效按键输入到视觉反馈出现的响应时间目标不超过 100 ms，最高不超过 300 ms。
- 连续进行高频输入时，页面不得出现明显卡顿、漏判或状态错乱。
- 在至少 100 次连续有效按键测试中，不应出现页面崩溃、无法继续练习或分数异常。
- 视觉设计必须满足儿童使用场景的基本可读性要求，包括大字号、高对比和清晰焦点区域。
- 反馈不应仅依赖颜色区分，需同时结合文字、图标或状态变化，降低误解风险。
- 核心可调参数应集中管理，至少包括轮次结束规则、反馈显示时长、是否启用声音反馈等，以降低后续改动成本。

## Risks

- 轮次结束规则未确认，会直接影响结果页设计、题目流转逻辑和验收口径。
- 是否加入音效未确认，若在开发后期新增，可能引入资源准备、开关逻辑和儿童环境噪音适配问题。
- 界面风格未确认，若在开发后期才决定具体方向，可能导致 UI 重做。
- 5 岁儿童存在注意力和精细动作差异，若界面元素过小或反馈不够直接，可能影响独立使用体验。
- `Caps Lock`、`Shift`、输入法状态或浏览器焦点问题可能导致按键体验异常，需要在实现中做容错。
- 不同家庭设备的键盘布局、浏览器版本和屏幕尺寸存在差异，可能影响一致性。
- 若错误后是否保留当前字母的规则在后续被修改，需同步调整反馈节奏和计分逻辑。

## Milestones

1. 需求锁定
   明确轮次结束规则、声音反馈是否纳入首版、界面风格方向，并冻结默认计分与错误处理规则。

2. 核心玩法可运行版本
   完成本地运行、开始页、单字母展示、键盘匹配、即时反馈、分数显示的最小闭环。

3. 交互打磨版本
   完成结果页、重新开始、焦点丢失提示、无效按键处理、儿童友好视觉优化。

4. 验收版本
   完成离线验证、桌面浏览器兼容性验证、连续输入稳定性验证和典型儿童使用流程走查。

## Acceptance Criteria

- 在断网状态下，通过本地打开网页后，用户可以进入并完成完整一轮练习。
- 首屏只提供清晰主入口，用户点击后进入练习状态，无需登录或联网。
- 练习中页面任一时刻只显示一个目标小写字母，且目标内容仅来自 `a-z`。
- 用户按下匹配字母键后，系统在可感知的即时范围内显示正确反馈，并使当前分数加 `1`。
- 用户按下不匹配字母键后，系统在可感知的即时范围内显示错误反馈，且在当前默认规则下分数不增加、当前字母不切换。
- 非字母按键不会导致页面报错、跳题或练习中断。
- 当达到本地配置的轮次结束条件时，系统进入结果页并显示本轮最终分数。
- 结果页提供明确的“再来一轮”入口，点击后可以开始全新一轮，且上一轮分数不保留。
- 刷新页面后，不存在历史记录、账号信息或上轮练习数据残留。
- 在基线浏览器环境下，连续完成至少 100 次有效按键输入，系统无崩溃、无卡死、无分数计算异常。
- 声音反馈不作为当前上线阻塞项；即使未实现声音，产品仍满足首版验收。
- 在轮次结束规则未最终确认前，验收以“系统可按配置完成一轮并正确结算”为准，不锁定具体题量或时长。
